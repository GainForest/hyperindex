## v0.5.0

v0.5.0 makes typed GraphQL filters more useful when clients already know the records or references they want. You can now batch hydrate records by AT-URI and filter inside common nested lexicon shapes without falling back to broad JSON search.

### For users

- Typed public record queries now include a metadata `uri` filter. Use `where: { uri: { eq: ... } }` for one known AT-URI, or `where: { uri: { in: [...] } }` to fetch a batch. The filter reads the stored record URI, not a `uri` key inside the JSON payload, so it still works when a lexicon has its own field named `uri`.
- Generated nested `where` filters now cover arrays, refs, and unions up to three lexicon path segments deep. This makes common relationship queries possible in GraphQL, such as finding collections whose `items` array contains a strong ref with a matching `itemIdentifier.uri`.
- Array `any` filters keep multiple predicates scoped to the same array item. A query that asks for both `itemIdentifier.uri` and `itemIdentifier.cid` only matches when both values are present on one array element, not split across different elements.
- Hypercert activity queries now include `contributorDid` for contributor matching. It covers inline contributor DIDs, legacy bare DID array entries, and `org.hypercerts.claim.contributorInformation` strong refs by checking the referenced record's `identifier`.

### Notes

- Nested scalar filters are exact-match only: `eq`, `in`, and `isNull`. Substring and range operators such as `contains`, `startsWith`, `gt`, and `lt` are intentionally not exposed for nested fields.
- Nested filters stay within the current record except for the activity-specific `contributorDid` compatibility filter. Use `search` or client-side filtering for deeper traversals or arbitrary JSON matching.
