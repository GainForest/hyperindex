## v0.7.0

v0.7.0 makes label filtering work at the author/account level, not just on individual records. Existing `externalLabels` filters still match labels attached to a record AT-URI. New `authorLabels` filters match labels attached to the record author's DID, which is the shape used for account-quality labels such as `standard`, `high-quality`, or `likely-test`.

### For users

- Typed record connection queries now accept `where.authorLabels` next to the existing `where.externalLabels`. Use `externalLabels` for labels on the record itself, and `authorLabels` for labels on the author DID.
- `authorLabels` supports the same `has` and `none` predicates as `externalLabels`, including `src`, `val`, and `activeOnly`. Clients can require authors with a trusted label or exclude authors with a known test/spam label without filtering pages on the client.
- Filtering runs before `edges`, cursors, `pageInfo`, and `totalCount` are calculated. The page and count you receive describe the filtered result set.

Example: exclude authors marked as likely test accounts by a specific labeler.

```graphql
where: {
  authorLabels: {
    none: {
      src: { eq: "did:plc:pswneepkd5lesumj7ejmkbal" }
      val: { eq: "likely-test" }
      activeOnly: true
    }
  }
}
```

### Notes

- `authorLabels` only matches DID-subject labels, for example `uri: "did:plc:..."` with no CID. It does not infer labels from profile records, organization records, or other records by the same author.
- Unlabeled authors pass `none` filters and do not pass `has` filters.
- There is no node-level `authorLabels` field. To display labels for known author DIDs, query `externalLabels(subjects: [...])`.
