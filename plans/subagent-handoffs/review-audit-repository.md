# Review — AuditRepository correctness

Scope reviewed: `internal/database/repositories/audit.go` and `internal/database/repositories/audit_test.go`, with cross-checks against `docs/tap-append-only-audit.md`, current migrations, and current `RecordsRepository`/`ActorsRepository` behavior.

## Blockers

- None found in the repository code path. The implementation is coherent enough for the next wiring layer, assuming GON-50 maps only already-validated `tap.Event` values into the repository-local `AuditTapEvent` type.

## Fixes worth doing now

- **Add an atomic rollback test before Tap ack wiring depends on this.** `IngestTapEvent` correctly opens one transaction, defers rollback, and commits only after raw, decoded, and current-state work succeeds (`internal/database/repositories/audit.go:206-224`). Raw rows are inserted before decoded rows (`internal/database/repositories/audit.go:334-342`, `377-385`), so a decoded insert/projection failure should leave no raw row and return an error that prevents ack. Current tests cover success/duplicates but not failure rollback (`internal/database/repositories/audit_test.go:33-98`). A small unsupported-action test can assert the CHECK failure rolls back `raw_tap_events` too.
- **Add tests for identity `is_active` missing vs explicit false.** The code intentionally stores missing `is_active` as SQL NULL (`internal/database/repositories/audit.go:501-505`) and distinguishes key slots as empty/`true`/`false` (`internal/database/repositories/audit.go:767-778`). Current identity tests cover present `true` and a purge with present `false` (`internal/database/repositories/audit_test.go:139-181`), but not missing `is_active` or a non-purge explicit false. This is exactly the adapter boundary GON-50 must preserve from `tap.IdentityEvent.IsActivePresent`.
- **Add a duplicate-projection immutability test.** The code only applies current-state updates inside `if inserted` (`internal/database/repositories/audit.go:347-365`, `390-400`), which matches the plan’s duplicate semantics. The current duplicate test replays the same event and checks row counts (`internal/database/repositories/audit_test.go:53-64`), but does not prove that a duplicate semantic key with a different body/CID-equivalent payload cannot overwrite current state. That test would protect the future handler from publishing/updating on duplicate decoded events.

## Optional / defer

- **Postgres repository execution still needs live DB coverage.** SQLite repository tests pass, and the Postgres raw payload fix is reflected by `payload TEXT NOT NULL` in the migration (`internal/database/migrations/postgres/007_add_audit_events.up.sql:7-13`) and no JSON cast in raw insert (`internal/database/repositories/audit.go:424-428`). The remaining Postgres-specific paths are still unexercised here: `record_events.record` casts to JSONB on insert (`internal/database/repositories/audit.go:462-464`), current `record.json` casts to JSONB (`internal/database/repositories/audit.go:577-583`), `record::text` scans (`internal/database/repositories/audit.go:275-285`), and timestamptz filters (`internal/database/repositories/audit.go:656-663`).
- **Repository-local event type relies on caller validation.** `IngestTapEvent` validates only nil/type/payload presence (`internal/database/repositories/audit.go:198-204`). Empty DID/collection/rkey/action strings would otherwise reach event-key construction and inserts; the audit table columns are NOT NULL but not non-empty (`internal/database/migrations/sqlite/007_add_audit_events.up.sql:29-34`). This is acceptable if GON-50 maps from `tap.ParseEvent` output, because Tap parsing validates the required record/identity fields, but the adapter should keep that invariant explicit.
- **`HasPreviousPage` is intentionally minimal.** `FindRecordEvents` sets `HasPreviousPage` to `opts.After != ""` (`internal/database/repositories/audit.go:324-329`). That is fine for forward cursor continuation, but it is not a full Relay previous-page calculation. GraphQL should document/use it accordingly unless backward pagination is added.

## Correct

- **Atomic transaction shape matches the plan.** The repository uses one transaction for ingest (`internal/database/repositories/audit.go:206-224`), inserts raw first, then decoded audit rows, then current projection only before commit (`internal/database/repositories/audit.go:334-410`), matching the planned transaction flow (`docs/tap-append-only-audit.md:238-261`).
- **Duplicate record semantics match the plan.** Every delivery inserts a new `raw_tap_events` row (`internal/database/repositories/audit.go:413-445`), decoded rows use `ON CONFLICT(event_key) DO NOTHING` (`internal/database/repositories/audit.go:466-489`), existing IDs are returned for duplicates (`internal/database/repositories/audit.go:494-498`), and current projection is skipped when `inserted` is false (`internal/database/repositories/audit.go:347-365`). Tests assert two raw rows but one decoded row for duplicate record delivery (`internal/database/repositories/audit_test.go:53-64`).
- **Record event keys follow the plan.** Normal keys use `record:{did}:{rev}:{collection}:{rkey}:{action}:{cid-or-empty}` (`internal/database/repositories/audit.go:760-765`), matching the source plan (`docs/tap-append-only-audit.md:97-114`). Missing rev falls back to delivery id plus normalized payload hash (`internal/database/repositories/audit.go:760-763`, `781-807`). Missing CID is stored as NULL while using the empty key slot (`internal/database/repositories/audit.go:448-456`, `760-765`), and query filtering treats empty CID as `'' OR NULL` (`internal/database/repositories/audit.go:693-699`).
- **Current-state record behavior mirrors existing repository semantics.** The audit upsert preserves the existing CID-same skip rule (`internal/database/repositories/audit.go:559-568`), matching `RecordsRepository.Insert` (`internal/database/repositories/records.go:147-160`), and uses equivalent SQLite/Postgres upsert SQL (`internal/database/repositories/audit.go:576-600`; existing SQL at `internal/database/repositories/records.go:169-187`). Deletes use the same URI/DID delete semantics (`internal/database/repositories/audit.go:603-644`; existing methods at `internal/database/repositories/records.go:965-976`, `internal/database/repositories/actors.go:195-201`).
- **Identity ingest matches current Tap behavior.** Identity keys include delivery id, DID, handle, is_active slot, and status (`internal/database/repositories/audit.go:767-778`), matching the first-version best-effort key (`docs/tap-append-only-audit.md:123-127`). Purge statuses delete current records and actor without synthetic record delete audit rows (`internal/database/repositories/audit.go:390-397`), matching the current-state plan (`docs/tap-append-only-audit.md:305-309`) and the test assertions (`internal/database/repositories/audit_test.go:170-191`).
- **Query API is safe for GraphQL basics.** User-provided filter values are parameterized (`internal/database/repositories/audit.go:647-710`), action values are validated (`internal/database/repositories/audit.go:680-686`), cursors are opaque base64 strings with a typed prefix (`internal/database/repositories/audit.go:832-850`), and ASC/DESC cursor comparisons are direction-aware (`internal/database/repositories/audit.go:253-264`).
