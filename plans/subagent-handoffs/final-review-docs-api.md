- Fixed: None. Review-only; no implementation or docs files were edited.
- Blocker: The README Tap quick-start command is not runnable with the checked-in Tap compose file because Hyperindex now requires `ADMIN_API_KEY`, but `docker-compose.tap.yml` does not pass it into the `hyperindex` service. The README tells operators to run `docker compose -f docker-compose.tap.yml up --build` (`README.md:73-82`), while the compose service only sets `HOST`, `PORT`, `TAP_*`, `DATABASE_URL`, `SECRET_KEY_BASE`, `EXTERNAL_BASE_URL`, `OAUTH_LOOPBACK_MODE`, and `ADMIN_DIDS` (`docker-compose.tap.yml:24-36`). Startup validation fails when `ADMIN_API_KEY` is missing (`internal/config/config.go:161-163`). Smallest safe fix: add `ADMIN_API_KEY: "${ADMIN_API_KEY:?ADMIN_API_KEY must be set - generate with: openssl rand -base64 32}"` to `docker-compose.tap.yml` and update the README Tap setup line to explicitly mention `ADMIN_API_KEY` alongside `TAP_ADMIN_PASSWORD` and `SECRET_KEY_BASE`.
- Note: The audit docs state the Tap ack happens after commit (`docs/tap-audit-mode.md:16-25`) and the README says ack-based delivery ensures no events are lost on crash (`README.md:67-70`), but `TAP_DISABLE_ACKS=true` disables acks (`README.md:110`; `.env.example:93-94`; `internal/tap/consumer.go:297-309`). Smallest safe fix: qualify those claims with “when `TAP_DISABLE_ACKS=false` (the default)”.
- Note: `TAP_ADMIN_PASSWORD` is described as only the Tap admin HTTP API password (`README.md:108-110`; `.env.example:87-91`), but Hyperindex also sends it as Basic auth for the Tap `/channel` WebSocket (`internal/tap/consumer.go:193-197`; wired from config at `cmd/hyperindex/main.go:812-817`). Smallest safe fix: describe it as the Tap admin/channel Basic auth password, used for both `/repos/add` and the WebSocket when Tap auth is enabled.
- Note: The “Continue after a cursor” example uses `after` without `orderBy` (`docs/tap-audit-mode.md:145-150`), but audit ordering defaults to DESC (`internal/graphql/schema/audit.go:94-100`; `internal/database/repositories/audit.go:819-822`). If the intent is “new events since the previous cursor”, this example should include `orderBy: { field: ID, direction: ASC }`; otherwise rename it to a plain “next page” example to avoid implying forward-since semantics.
- Note: The duplicate-delivery docs say semantic record events are deduped by DID, revision, collection, rkey, action, and CID (`docs/tap-audit-mode.md:33-37`), but events with missing `rev` use a fallback key based on Tap delivery id and normalized payload hash (`internal/database/repositories/audit.go:760-764`). Smallest safe fix: add one sentence noting missing-rev events use best-effort fallback dedupe.
- Note: README says audit mode stores “record and identity history” and then says audit history is available through `auditRecordEvents` (`README.md:113-117`). The GraphQL API currently exposes record audit rows only (`internal/graphql/schema/builder.go:458-463`; `internal/graphql/schema/audit.go:103-175`). Smallest safe fix: say `auditRecordEvents` exposes record history, while `raw_tap_events` and `identity_events` are stored in the database but do not have a public GraphQL query in this first API.
