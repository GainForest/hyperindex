Hyperindex query primitive

recordTimeline without product coupling

Interactive model for a generic GraphQL resolver that returns current records from selected collections, filtered by author DIDs, ordered by materialized createdAt, paginated with one stable cursor, and optionally batch-hydrated with Certified profile data.

Core idea

The app resolves “who I follow” separately, then asks Hyperindex for one merged page across the collections it cares about. Hyperindex stays generic; optional profile hydration improves Hypercerts-stack DX without changing feed policy.

Safety model

              
Why this can be safe, and where it becomes expensive.

            

            illustrative estimate
          

          

            

              Safe path
              Materialize record creation time at ingest. Query with bounded author and collection lists. Hydrate profiles only after pagination.
            

            

              Expensive path
              Extract createdAt from JSON for every candidate row and sort it at request time.
            

            

              Non-goal
              This is not a follow resolver, typed union resolver, or append-only event log. Updates/deletes need a separate recordEvents API.
